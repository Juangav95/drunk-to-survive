# Technical Specification — Drunk to Survive

This document is the detailed design behind the [README](../README.md). It's the
single source of truth for *how* the tracker works, so we can build it in phases
without losing the thread.

---

## 1. Goal

A static, F1-themed website that visualizes a season-long head-to-head contest
between **Gav** and **Nico** across the 2026 F1 World Championship, updated from
official results with a single click (and optionally automatically).

---

## 2. Contest Rules (authoritative)

- Two contenders: **Gav** and **Nico**.
- Each picks **3 drivers** and **3 teams** before the season.
- For each race weekend:
  - `driver_points` = sum of the official points scored by the contender's 3 drivers
    (Grand Prix points + Sprint points).
  - `team_points` = sum of the official constructor points scored by the
    contender's 3 teams (Grand Prix + Sprint).
  - `weekend_points = driver_points + team_points`.
- `season_total` = running sum of `weekend_points` across all completed weekends.
- Highest season total at the end of 2026 wins.

### Picks (canonical IDs resolved during Phase 2)

| Contender | Drivers | Teams |
|---|---|---|
| **Gav** | Antonelli, Piastri, Hadjar | Mercedes, Red Bull, Williams |
| **Nico** | Hamilton, Antonelli, Colapinto | Ferrari, Alpine, Cadillac |

**Edge cases to honor:**
- **Shared pick:** Antonelli is on both rosters; his points count for both.
- **Driver+team double-count:** A contender may hold both a driver *and* that
  driver's team (e.g. Nico has Hamilton + Ferrari). This is intentional — the
  driver's points count toward the driver tally, and the team's constructor
  points (which include that driver) count toward the team tally. We do **not**
  de-duplicate; that's the agreed ruleset.

## 3. Scoring Reference (official F1)

| Position | GP points | Sprint points |
|---:|---:|---:|
| 1 | 25 | 8 |
| 2 | 18 | 7 |
| 3 | 15 | 6 |
| 4 | 12 | 5 |
| 5 | 10 | 4 |
| 6 | 8 | 3 |
| 7 | 6 | 2 |
| 8 | 4 | 1 |
| 9 | 2 | 0 |
| 10 | 1 | 0 |

- Points outside these positions: 0.
- We read **awarded points directly from the API** rather than recomputing from
  finishing position, so any sporting adjustments (penalties, DSQs) are reflected.
- A **fastest-lap bonus point does not apply** under current rules (removed for 2025+).

## 4. Data Source — Jolpica-F1 API

Base URL: `https://api.jolpi.ca/ergast/f1/`

Endpoints we'll use (JSON):
- **Season schedule:** `…/2026/races/?format=json` → ordered list of rounds + names.
- **Race results:** `…/2026/{round}/results/?format=json` → per-driver points & constructor.
- **Sprint results:** `…/2026/{round}/sprint/?format=json` → per-driver sprint points.
- (Constructor points per weekend are derived by summing each constructor's
  drivers' points from the results above — equivalent to official constructor scoring.)

**Verification tasks at build time:**
1. Confirm 2026 data is being published and how promptly after each weekend.
2. Resolve canonical Jolpica IDs for each pick:
   - Drivers: `antonelli`/`kimi_antonelli`, `piastri`, `hadjar`, `hamilton`, `colapinto` (exact IDs TBD).
   - Constructors: `mercedes`, `red_bull`, `williams`, `ferrari`, `alpine`, and
     **Cadillac** (new 2026 entry — confirm the exact constructor ID).
3. Handle rounds not yet run (no results) gracefully.

**Resilience:** results are persisted to `data/results.json` in the repo, so the
site never depends on the API being live at view time.

## 5. Data Model

### `data/config.json` (hand-authored, stable)
```json
{
  "season": 2026,
  "contenders": [
    {
      "id": "gav",
      "name": "Gav",
      "photo": "assets/img/gav.jpg",
      "color": "#00D2BE",
      "drivers": ["antonelli", "piastri", "hadjar"],
      "teams": ["mercedes", "red_bull", "williams"]
    },
    {
      "id": "nico",
      "name": "Nico",
      "photo": "assets/img/nico.jpg",
      "color": "#E8002D",
      "drivers": ["hamilton", "antonelli", "colapinto"],
      "teams": ["ferrari", "alpine", "cadillac"]
    }
  ]
}
```

### `data/results.json` (generated by the Action)
```json
{
  "generated_at": "2026-06-22T16:00:00Z",
  "season": 2026,
  "rounds": [
    {
      "round": 1,
      "name": "Australian Grand Prix",
      "short": "AUS",
      "date": "2026-03-08",
      "completed": true,
      "scores": {
        "gav":  { "drivers": 33, "teams": 40, "weekend": 73, "cumulative": 73 },
        "nico": { "drivers": 28, "teams": 31, "weekend": 59, "cumulative": 59 }
      }
    }
  ]
}
```

The site reads `config.json` + `results.json` and needs nothing else at runtime.

## 6. Frontend

- **Stack:** plain HTML/CSS/JS, no build step (keeps GitHub Pages trivial).
- **Chart:** Chart.js line chart for the cumulative trace.
  - X-axis: Grand Prix short names in round order.
  - Y-axis: cumulative total points.
  - One line per contender, colored by `config.color`.
  - Only `completed: true` rounds are plotted.
- **Components:**
  1. **Contender bubble** — photo, name, live total, in F1 livery.
  2. **The trace** — the headline cumulative line chart.
  3. **Standings table** — totals, gap, driver vs team split.
  4. **Per-race breakdown** — expandable per-round detail.
- **Theme:** F1-styled — bold type, team colors, dark background, driver/team cards.

## 7. Data Pipeline (GitHub Action)

File: `.github/workflows/update-results.yml`

- **Triggers:**
  - `workflow_dispatch` — the one-click "Run workflow" button.
  - `schedule` (cron) — e.g. Sunday/Monday evenings during the season for auto-updates.
- **Steps:**
  1. Check out the repo.
  2. Run `scripts/fetch_results.py`:
     - Read `data/config.json`.
     - Pull schedule + race + sprint results for 2026 from Jolpica.
     - Compute per-weekend and cumulative scores per contender.
     - Write `data/results.json`.
  3. Commit & push `data/results.json` if it changed.
- **Permissions:** the workflow needs `contents: write` to commit results.
